## Project Overview

Molq is a unified job queue system for local and HPC cluster execution. It provides a `Cluster` destination type plus an instance-based `Submitor` engine for submitting, monitoring, and managing computational jobs across different scheduler backends (local, SLURM, PBS, LSF).

## Architecture

### Core Design: Submitor + Scheduler + Store

Three-layer architecture:

1. **Public API** (`cluster.py`, `submitor.py`, `__init__.py`):
   - `Cluster(name, scheduler)` is the destination spec — pure config, no lifecycle state
   - `Submitor(target=cluster)` is the lifecycle engine, bound to a target `Cluster`
   - `JobHandle` returned by `submit_job(...)` for single-job operations; `JobHandle.status()` and `JobHandle.wait()` for tracking
   - All public types are frozen dataclasses (`Memory`, `Duration`, `Script`, `JobResources`, etc.)

2. **Scheduler Layer** (`scheduler.py`):
   - `Scheduler` Protocol: `capabilities()`, `submit()`, `poll_many()`, `cancel()`, `resolve_terminal()`, `list_queue()`
   - Implementations: `ShellScheduler` (the backend for `scheduler="local"` — no batch system, transport decides where), `SlurmScheduler`, `PBSScheduler`, `LSFScheduler`
   - All four implementations route every shell call through `self._transport.run(...)`, so any `Scheduler × Transport` combination works
   - Script generation absorbed into each scheduler (not a separate protocol)

3. **Persistence Layer** (`store.py`):
   - `JobStore`: SQLite with WAL mode, UUID-based `job_id`, schema versioning
   - `molq_meta` table tracks schema version; auto-migrates from v1
   - Reconciler (`reconciler.py`) syncs scheduler state with DB

4. **Monitoring Layer** (`monitor.py`, `reconciler.py`):
   - `JobReconciler`: batch-queries scheduler, computes diffs, updates store
   - `JobMonitor`: blocking wait with pluggable `PollingStrategy`
   - `EventBus` (`callbacks.py`): pub/sub for job lifecycle events

5. **CLI** (`cli/main.py`): Typer + Rich — commands: submit, list, status, logs, watch, history, inspect, monitor, cancel, cleanup, daemon

### Module Dependency Graph

```
Layer 0: errors.py, status.py           (enums, exceptions)
Layer 1: types.py                        (Memory, Duration, Script, JobResources, etc.)
Layer 2: options.py                      (SchedulerOptions types)
Layer 3: models.py                       (Command, JobSpec, JobRecord, SubmitorDefaults)
Layer 4: merge.py                        (defaults merge function)
Layer 5: store.py                        (SQLite persistence)
Layer 6: scheduler.py                    (Scheduler protocol + implementations)
Layer 7: reconciler.py                   (scheduler ⇄ DB sync)
Layer 8: strategies.py, callbacks.py     (polling strategies, event bus)
Layer 9: monitor.py                      (polling engine)
Layer 10: cluster.py                     (Cluster destination spec)
Layer 11: submitor.py                    (Submitor + JobHandle)
Layer 12: __init__.py                    (public re-exports)
Layer 13: cli/main.py                    (Typer CLI)
```

### Key Patterns

- **Protocol-based schedulers**: `Scheduler` is a `typing.Protocol` — new backends implement `capabilities()`, `submit()`, `poll_many()`, `cancel()`, `resolve_terminal()`, `list_queue()`
- **Frozen dataclasses**: All public types are `@dataclass(frozen=True)` — immutable, no Pydantic
- **Structured errors**: All exceptions inherit `MolqError`, carry typed context attributes
- **UUID job identity**: `job_id` is a UUID v4 generated by molq; scheduler-assigned IDs stored as `scheduler_job_id`
- **Zero import side effects**: `import molq` creates no files, opens no DB

### Adding a New Scheduler Backend

1. Add a new class implementing the `Scheduler` protocol in `scheduler.py`
2. Add a corresponding `SchedulerOptions` frozen dataclass in `options.py`
3. Register in `OPTIONS_TYPE_MAP` in `options.py`
4. Add factory branch in `create_scheduler()` in `scheduler.py`

### Command Model

Three mutually exclusive command types:
- `argv: list[str]` — structured args, never shell-interpreted
- `command: str` — single-line shell command (no newlines)
- `script: Script` — `Script.inline(text)` or `Script.path(path)`

## Testing Conventions

- Fixtures in `conftest.py`: `temp_workdir`, `tmp_molq_home`, `mock_job_environment`
- JobStore tests use `:memory:` SQLite or `tmp_path` file databases (no mocking SQLite)
- Scheduler tests mock `molq.transport.subprocess.run` (the call site for every Scheduler/Transport combination)
- Submitor tests mock the Scheduler protocol via `unittest.mock.MagicMock`
- Script.path tests use `tmp_path` with real files
